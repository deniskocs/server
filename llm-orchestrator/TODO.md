# LLM Orchestrator — заметки / дорожная карта

Краткое место, **на чём остановились** и **что делать дальше**, чтобы при следующем заходе в репо было понятно направление.

## Текущее состояние (сейчас)

Оркестратор **в рабочем виде** и закрывает **минимальную** потребность:

- старт / стоп vLLM через Docker;
- загрузка весов (Hugging Face) в `MODELS_DIR`;
- базовый UI: таблица конфигов, статусы, действия, блок host-stats (CPU/RAM/GPU, размер `MODELS_DIR` и свободное место на томе), прогресс скачивания на бэке + опрос `GET /models` при длинном `POST` download.

Инфра сейчас ориентирована на **одну хост-машину** как единую точку (API + сокет Docker + веса).

## Ближайшие улучшения (код / UX)

### 1. Несколько моделей параллельно — имена контейнеров

- **Сделано:** имя контейнера = `vllm-orchestrated--<config-stem>--<хвост DEFAULT_MODEL_NAME/…>` (см. `vllm_container_name` в `vllm_env.py`); `start` снимает только свой контейнер, `stop` — по `configFile`.
- Liveness по-прежнему по `PORT` из `.env`.
- После обновления со старого `vllm-orchestrated` на хосте может остаться **один** legacy-контейнер — останови вручную `docker stop vllm-orchestrated` при миграции.

### 2. UI: «Running» только после готовности; загрузка = Progress, не «кнопка Loading»

- **Start:** после нажатия показывать стейт **Running / ожидание liveness** и переключать в **«модель активна»** только когда **проходит** проверка (например `GET /v1/models` на `PORT` из `.env`), а не сразу по окончании `docker run`.
- **Download:** вместо ощущения «висит кнопка Loading» — явно **Progress** (или скрытие кнопки) и уже существующий **прогресс %** / индетерминатно, не перегружая текст.
- Уже частично есть прогресс на бэке; нужно **согласовать** с copy и переходами состояний в таблице.

### 3. Рефреш слишком тяжёлый; периодический «лёгкий» sync

- Полный ручной refresh **долгий/тяжёлый**.
- **Идея:** фоновое обновление состояния (хотя бы **liveness + статус скачивания**) с периодом **~10 с**, без полного перерисывания всей страницы при каждом тике (или с лёгким diff).
- Может быть: отдельный лёгкий endpoint или тот же `GET /orchestrator/models` с throttling на клиенте + оптимизация на бэке (не делать `du` / тяжёлые вещи на каждый poll).

## Средний срок: дополнительные «раннеры» (отдельные сервисы)

Не vLLM-LLM, а **отдельные** образы/процессы за тем же оркестратором (или рядом):

1. **Embeddings** (отдельный раннер).
2. **Генерация изображений** (отдельный раннер).
3. **Генерация звука (TTS)**.
4. **Audio → text** (ASR / whisper-подобное).

Пока **идеи уровня архитектуры**: отдельные Docker-имена, порты, health, те же `*.env` или отдельный тип конфига — **не зафиксировано**.

## Дальше: распределение и типы нод

- Сейчас: **одна** хост-машина, единая точка обслуживания.
- **Цель:** оркестратор по-прежнему может жить **на одной** машине, но конфигурация **«orchestrated node(s)»** (рабочее имя): узлы **разных типов** (например **Linux / macOS**), куда **делегируются** run/pull/модели.
- Требуется проектирование: реестр нод, credentials, сеть, **где** крутится Docker (локально / SSH / API).

---

## API: агрегатор в стиле OpenAI

- **Цель:** единая **OpenAI-совместимая** поверхность (например `/v1/models`, chat), которая:
  - **собирает** модели **со всех** подключённых сервисов (vLLM, будущие раннеры, возможно внешние endpoint’ы);
  - при обращении **маршрутизирует** (redirect / reverse proxy / выбор бэкенда) на нужный сервис по имени модели.
- Это **отдельный** слой над текущим FastAPI оркестратора или отдельный gateway — **TBD**.
