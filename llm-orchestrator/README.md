# LLM Orchestrator

Сервис и Docker-образ для управления LLM на сервере: конфигурация моделей, загрузка/удаление, запуск/остановка, небольшой веб-интерфейс.

Статус: **черновик требований** — основная спецификация здесь. В [`web/`](web/) — UI (Vite + TypeScript), **Docker-образ** (nginx + `dist/`), **ручной deploy на Mac**: [`.github/workflows/deploy-orchestrator.yaml`](../.github/workflows/deploy-orchestrator.yaml) (**Deploy Orchestrator**). **Бэкенд** **ещё** не реализован.

## Что планируется в этой папке

| Артефакт | Назначение |
|----------|------------|
| **Docker-образ** | Упаковка оркестратора: веб-часть + логика работы с конфигом и жизненным циклом моделей. |
| **Скрипт релиза / установка (web)** | Сборка и публикация образа + SSH на Mac: [`deploy-orchestrator.yaml`](../.github/workflows/deploy-orchestrator.yaml) (теги и порты — в yaml), отдельно от deploy router. |

## Архитектура деплоя

- После **деплоя** на сервере поднимаются **два** сервиса (два контейнера) в **одной** отдельной user-defined **Docker network** — не в сети других проектов (например не в `learn_english` у learn-english, а **своя** сеть под этот продукт).
- **Имя сети** задать осмысленно, в духе оркестратора, например: **`llm_orchestrator`** (или `orchestrator_net` / `orchestrator` — окончательно при `docker network create` / compose; в логике: «Orchestrator», не «Learn English»).
- **Nginx** — контейнер со **статикой** (собранный фронт: `index.html`, ассеты из `dist/`), `proxy_pass` к API бэкенда, при необходимости TLS, gzip, маршруты.
- **Сервис оркестратора** — **бэкенд**: API, чтение/запись каталога конфигов, оркестрация Docker/моделей и остальное по спецификации UI. Nginx стучится к бэку по **внутреннему** имени сервиса в сети; порт бэка **снаружи** можно не публиковать.
- **Тома** (каталог конфигов, при необходимости `docker.sock` или путь к хосту) — на **бэкенд-контейнер**; Nginx, как правило, **без** этих томов, кроме собственной **статики** (копия в образе или смонтированный `dist`).

## Конфигурация (связь с `vllm/llm-configs`)

- Эталон в репозитории: папка **[`vllm/llm-configs/`](../vllm/llm-configs/)** (путь **от корня проекта** — `vllm` → `llm-configs`). Там, например, шаблоны `.env` для vLLM. При обсуждении «llm-config» без уточнения в этом README речь про **эту** директорию.
- Разбор YAML и примеры профилей, если понадобятся при реализации, можно согласовать с отдельным корневым каталогом [`llm-configs/`](../llm-configs/) в репо и с [`load_config.py`](../llm-configs/load_config.py) — внутри `vllm` своего `load_config` сейчас нет, это опора на соседний по смыслу код.
- На сервере в **одной выделенной папке** (volume / bind mount) лежат **все файлы конфигов** рантайма; путь к монтированию зададим, когда появятся бинарь/образ. Оркестратор читает и изменяет только эту директорию.
- При проектировании UI решим, что дублируем из репо, а что остаётся только в compose/deploy.

## Веб-интерфейс (минимальный)

### Список записей (каждая строка)

Таблица или список строк **по выбранному конфигу** (в духе [`vllm/llm-configs`](../vllm/llm-configs/)). В одной строке слева направо:

| Зона | Содержимое |
|------|------------|
| **№** | Порядковый номер строки (1, 2, 3, …). |
| **Имя** | Человекочитаемое имя записи / идентификатор (как в конфиге). |
| **Конфигурация модели** | Краткое отображение ключевых параметров этой записи (модель, квантизация, контекст и т.п. — что возьмём из полей конфига; детали при вёрстке). |
| **Кнопки (справа)** | Набор действий **зависит от состояния** этой записи: загрузка модели (с прогрессом), удаление модели с диска, **Play** / **Stop**, при необходимости «удалить конфиг-файл» на уровне файла — как в разделе ниже; неактивные или лишние кнопки не показываем или дизейблим. |

Простая, аккуратная вёрстка: выравнивание колонок, на мобильном можно стэк или горизонтальный скролл — уточним при реализации.

### Под списком — добавление конфига

- Заметная зона: **добавить новый конфиг** — кнопка вроде «Add new config» и/или **плюсик** (⊕ / +).
- По нажатию открывается **окно** (модалка, боковая панель и т.п. — деталь реализации), в нём:
  - **Имя** — задаёт **имя файла** конфига (как оно попадёт в каталог на диске; расширение, если нужно, — по соглашению при реализации).
  - **Текст конфига** — многострочное поле, куда пользователь **вставляет** готовый текст (или набирает); это содержимое файла.
  - **Save** — сохраняет файл в каталог конфигов и **сразу** добавляет соответствующую строку в **ту же таблицу** (без перезагрузки страницы), чтобы новая запись была видна сразу после успешного сохранения.

### Операции (логика кнопок)

- Работа с **каталогом конфигов** на диске (все файлы конфигов лежат **только** в этой папке):
  - **Добавить конфиг** — через **плюс** → окно с **именем файла** + **текстом** + **Save**; после Save файл на диске и новая строка в таблице (см. «Под списком»).
  - **Удалить конфиг** — удалить выбранный конфиг-файл из папки.  
  *(Это про файлы настроек; снятие **модели** с диска — отдельная операция.)*
- **Для каждой записи (в рамках конфига):**
  - **Загрузить модель** на сервер (если ещё нет) — в процессе **прогресс загрузки**.
  - **Удалить модель** с диска сервера.
  - Если модель загружена — **Play** по строке/блоку конфига; **Stop** — остановка инстанса.
- При **запуске (run)** — **результат/статус** последнего выполнения (успех, ошибка, краткое сообщение — контракт позже).

## Фронтенд: стек и production-сборка

### Стек

- **HTML**, **CSS**, **JavaScript** в браузере; исходники логики UI предпочтительно писать на **TypeScript** (строгие типы, тот же выход в JS после сборки). Конкретные фреймворки (только ваниль, лёгкий шаблон и т.д.) — решим при старте кода, но база: классическая тройка + TS.

### Как в современных проектах устраивают «бандл» и отдачу в проде

В **разработке** обычно крутят **дев-сервер** (например, на базе **Vite** или аналога): TypeScript и CSS **на лету** превращают в то, что понимает браузер, с быстрой перезагрузкой. Это **не** то, что кладут на бой.

**Production-сборка** — отдельная команда (условно `npm run build`):

1. **TypeScript** компилируется в **JavaScript** (проверка типов, один или несколько `.js` файлов).
2. **Связка (bundler)** — один инструмент (часто **Vite** под капотом **esbuild** / **Rollup**, либо **webpack**): собирает модули в **финальные бандлы**, выкидывает неиспользуемый код (**tree-shaking**), при необходимости режет на **чанки** (чтобы кэшировать отдельно).
3. **Минификация**: имена сокращаются, пробелы и переводы строк убираются, выражения уплотняются — это **Terser**, **esbuild minify** и т.п. В проде в логе «обфусцированный жаргон» в основном = **такой сжатый и переписанный минифицированный JS**; отдельный шаг **obfuscator** (целенаправленная запутывающая обфускация) делают реже и только если есть требования к усложнению чтения — **не** обязательный стандарт; для внутреннего admin-UI обычно достаточно **minify + no source maps in prod** (или source maps только приватно).

4. **CSS** тоже минифицируется и, при желании, из **PostCSS/Autoprefixer** в зависимости от договора.

5. **Результат** — каталог вроде `dist/` или `build/`: `index.html`, **хэш в имени файла** у JS/CSS (`app-a1b2c3.js`) для **кэширования** «навсегда» у CDN/браузера.

**Деплой в образ оркестратора:** в **Docker-образ** кладут **уже** собранный `dist/` (копируем на этапе `docker build` или в CI до сборки образа). Внутри контейнера **один HTTP-процесс** (тот же бинарь бэкенда, что отдаёт API) раздаёт эти файлы как **статику** (например, по пути `/` и `/assets/...`) на **нужном порту**; либо перед ним **nginx** — как решите в `Dockerfile` / compose. Снаружи открыт **тот порт**, который пробрасываем при `docker run` / в workflow установки. Пользователю важно: **в прод идёт не сырой TS**, а **готовая статика** из бандла.

### Кратко

| Этап | Что происходит |
|------|----------------|
| `dev` | TS/CSS → dev-сервер, удобная отладка. |
| `build` | TS → JS, бандл, **minify**, выход в `dist/`. |
| Образ | `dist/` внутри контейнера, раздача по HTTP на **заданном порту**. |

Точные имена `package.json`, lockfile и выбор Vite vs другой — зафиксируем в репозитории при появлении кода.
